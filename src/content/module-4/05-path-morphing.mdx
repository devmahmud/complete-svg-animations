# Path Morphing

<Objective>
- Animate `d` to morph one shape into another
- Apply the **same-command-count rule** so morphs interpolate smoothly
- Pad a simpler shape so it has the same structure as a more complex one
</Objective>

Animating the `d` attribute itself is the holy grail — shapes flow into other shapes. CSS can't do this; SMIL can. There's exactly one rule to make it work: **every keyframe's path must have the same number of commands** in the same order.

## The simplest morph

<LiveSVG code={`<svg viewBox="0 0 100 100">
  <path fill="#7c5cff" stroke="#ffd43b" stroke-width="2">
    <animate attributeName="d"
             values="M 20 20 L 80 20 L 80 80 L 20 80 Z;
                     M 50 20 L 80 50 L 50 80 L 20 50 Z;
                     M 20 20 L 80 20 L 80 80 L 20 80 Z"
             dur="3s" repeatCount="indefinite"/>
  </path>
</svg>`} caption="Square ↔ diamond. Both have 4 L commands + Z." />

## Why command count matters

If keyframes have different command counts, the renderer **snaps** instead of interpolating — it can't tell which point should slide to which. Always pad shorter shapes:

<LiveSVG code={`<svg viewBox="0 0 100 100">
  <!-- A "line" expressed as a degenerate quad so it has the same shape as the box -->
  <path fill="#7c5cff" stroke="#ffd43b" stroke-width="2">
    <animate attributeName="d"
             values="M 20 50 L 80 50 L 80 50 L 20 50 Z;
                     M 20 20 L 80 20 L 80 80 L 20 80 Z;
                     M 20 50 L 80 50 L 80 50 L 20 50 Z"
             dur="3s" repeatCount="indefinite"/>
  </path>
</svg>`} caption="The 'line' is really a 4-point path with two pairs collapsed — same command shape as the box, so the morph interpolates smoothly." />

## Curves morphing

Swap `L` for `C` (cubic) for organic transitions. Make sure the cubic command count matches across keyframes.

<LiveSVG code={`<svg viewBox="0 0 100 100">
  <path fill="#3ddc97">
    <animate attributeName="d"
             values="M 20 50 C 20 30, 80 30, 80 50 C 80 70, 20 70, 20 50 Z;
                     M 20 50 C 30 20, 70 20, 80 50 C 70 80, 30 80, 20 50 Z;
                     M 20 50 C 20 30, 80 30, 80 50 C 80 70, 20 70, 20 50 Z"
             dur="3s" repeatCount="indefinite"/>
  </path>
</svg>`} caption="Two C commands per keyframe → smooth blob breathing." />

## Direction matters

The path's "starting point" travels with it. If you want a triangle to flip without spinning, make sure the points go **the same way around** in both keyframes.

<LiveSVG code={`<svg viewBox="0 0 100 100">
  <path fill="#ffd43b">
    <animate attributeName="d"
             values="M 50 20 L 80 80 L 20 80 Z;
                     M 50 80 L 80 20 L 20 20 Z;
                     M 50 20 L 80 80 L 20 80 Z"
             dur="2s" repeatCount="indefinite"/>
  </path>
</svg>`} caption="Triangle pointing up → down → up. All keyframes go clockwise." />

<Callout type="tip" title="Keep complex morphs in keyTimes">
  Use the <code>keyTimes</code> attribute to pace specific transitions. <code>keyTimes="0; 0.7; 1"</code> spends 70% of the duration morphing one direction and 30% returning.
</Callout>

## When to reach for a library

Hand-authored morphs work for 2–5 keyframes. Beyond that, use a library like Flubber or Anime.js to interpolate between arbitrary paths — they automatically pad and reorder commands so any two paths can morph into each other.
