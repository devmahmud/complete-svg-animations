# The Problem with CSS

<Objective>
- Identify the SVG attributes CSS cannot animate
- Decide between CSS and SMIL for any given task
- Understand why both systems exist and when to mix them
</Objective>

CSS gives you `transition` and `@keyframes`, plus the GPU-friendly `transform` and `opacity`. That's enough for **most** UI. But SVG has attributes CSS can't reach — and that's where SMIL earns its place.

## What CSS can animate on SVG

- `fill`, `stroke`, `stroke-width`, `stroke-opacity`
- `opacity`
- `transform` (translate, scale, rotate, skew)
- `r`, `cx`, `cy`, `x`, `y`, `width`, `height` (modern browsers — but layout-bound)
- `stroke-dasharray`, `stroke-dashoffset`

## What CSS cannot animate

- **`d`** — the path data string (the heart of every shape)
- **`points`** — polyline / polygon point lists
- **`viewBox`** — the SVG camera
- **Gradient stops** (`offset`, `stop-color`, `stop-opacity`)
- **Anything along a path** — there's no CSS equivalent of `<animateMotion>`

## When to pick which system

| Need | CSS | SMIL |
| --- | --- | --- |
| Hover / focus state changes | ✅ best | ⚠️ awkward |
| Looping ambient motion | ✅ | ✅ |
| Animating `transform` / `opacity` | ✅ best (GPU) | ✅ |
| Animating `d`, `points`, `viewBox` | ❌ no | ✅ only option |
| Following an arbitrary path | ❌ partial via `offset-path` | ✅ best |
| Synchronizing multiple animations on events | ⚠️ JS-glue | ✅ built-in |
| Internet Explorer support | ✅ | ❌ |
| GPU layer, 60fps with hundreds of elements | ✅ | ⚠️ depends |

<Callout type="tip" title="Reach for SMIL when CSS hits a wall">
  In production code, mix freely: CSS for state changes and ambient layouts, SMIL for the things only SMIL can do. There's no penalty for combining.
</Callout>

## What SMIL is

**SMIL** = Synchronized Multimedia Integration Language. In SVG you'll only ever use four elements: `<animate>`, `<animateTransform>`, `<animateMotion>`, and `<set>`. Each one lives **inside** the shape it animates. The next lesson introduces them.
